## 1. 课题背景

### 1.1 选题来源

本课题来源于成都信息工程大学《工程实践4》课程的教学实际需求。课程面向软件工程专业研究生，要求学生在16周内以团队协作方式完成一个完整的 Web 软件产品。班级采用 GitHub 进行团队协作开发，45 人同时向同一仓库提交代码，确保 commit 消息格式的一致性成为保障代码历史清晰可追溯的必要手段。

### 1.2 当前存在的痛点

1. **Commit 消息格式不统一**：团队成员提交 commit 时使用不同的格式风格，有的用中文、有的用英文、有的无说明，导致 Git 历史难以阅读和追溯。

2. **缺乏自动化校验手段**：没有工具在 commit 提交时自动检查消息格式，不规范的消息直到 Code Review 时才被发现，增加审查成本。

3. **CI/CD 流程中缺少 Commit 质量门禁**：已有的 CI 自动化（F-D-004）仅检查代码风格和单元测试，未覆盖 commit 消息质量，无法在 PR 层面阻断不规范提交。

### 1.3 相关文献与开源项目

[1] Conventional Commits. *A specification for adding human and machine readable meaning to commit messages*. https://www.conventionalcommits.org, 2025.

[2] Marshmallow Code. *commitlint: Lint commit messages*. https://commitlint.js.org, 2025.

[3] GitHub, Inc. *GitHub Actions Documentation — CI/CD & Automation*. https://docs.github.com/en/actions, 2025.

[4] Reflex Development Team. *Reflex: Web Apps in Pure Python*. https://reflex.dev, 2025.

## 2. 课题目标（SMART 原则）

### S（Specific，具体）

实现 Commit 消息格式自动校验功能（F-D-009）：在平台的「开发运维」页面提供 commitlint 配置向导，支持在线编辑规则并一键生成 CI 配置文件提交至仓库，不合规 commit 将导致 CI 失败并阻断 PR 合并。

### M（Measurable，可量化）

| 指标 | 目标值 |
|------|--------|
| 配置向导功能完成度 | 规则集选择 + 参数编辑 + 保存全部可用 |
| 配置文件生成准确率 | 生成的 `.commitlintrc.json` 和 `commitlint.yml` 格式正确 |
| TDD 测试通过率 | 5/5 全部通过 |
| CI 联动 | 启用时不合规 commit 阻断 PR，停用时跳过 |

### A（Achievable，可实现）

使用 Python（FastAPI）开发后端 API，commitlint 为成熟的 npm 工具，有完善的 GitHub Actions 集成方案，GitHub Contents API 可完成配置文件推送。

### R（Relevant，相关）

该功能直接解决团队协作中 commit 消息格式不统一的问题，与 Git 工作流和 CI 自动化（F-D-004）紧密联动，属于 DevOps 工具链的重要组成部分。

### T（Time-bound，有时限）

| 阶段 | 时间节点 | 交付物 |
|------|---------|--------|
| 基础框架搭建 | 第10周 | FastAPI 路由 + 数据库表结构 |
| 核心功能实现 | 第11-12周 | 配置向导 CRUD + 配置文件生成引擎 |
| GitHub 推送集成 | 第13周 | Contents API 推送 + 分支选择器 |
| 测试与完善 | 第14周 | TDD 测试通过 + 前端界面完善 |

## 4. 功能简述（F-D-009 范围）

本人在团队中负责的需求 F-D-009 包含以下子功能：

| 编号 | 功能 | 简要描述 |
|------|------|---------|
| F-D-009-01 | 配置向导 | 选择规则集（Conventional Commits 标准 / 自定义），在线编辑 type 枚举、header 最大长度、subject 最小长度 |
| F-D-009-02 | 配置文件生成 | 一键生成 `.github/workflows/commitlint.yml` 与 `.commitlintrc.json` |
| F-D-009-03 | 仓库提交 | 将配置文件通过 GitHub Contents API 推送至仓库，纳入 Git 版本控制 |
| F-D-009-04 | CI 联动 | 启用时规则级别为 error（阻断 PR），停用时为 0（跳过校验） |
| F-D-009-05 | 状态展示 | 显示启用/停用状态、规则版本、最近 5 次校验失败记录 |
| F-D-009-06 | 分支选择 | 下拉框选择目标分支，灵活指定推送位置 |
