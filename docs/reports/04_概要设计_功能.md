### 3.1 Commitlint API 列表

| 方法 | 路径 | 功能 | 输入 | 输出 |
|------|------|------|------|------|
| GET | /api/teacher/commitlint/config | 获取配置 | — | 完整配置对象 |
| PUT | /api/teacher/commitlint/config | 更新配置 | 规则字段 | 更新后的配置 |
| POST | /api/teacher/commitlint/toggle | 切换启用 | {enabled: bool} | {enabled: bool} |
| GET | /api/teacher/commitlint/status | 状态总览 | — | 配置 + 失败记录 |
| GET | /api/teacher/commitlint/failures | 失败记录 | — | 最近 5 条失败列表 |
| POST | /api/teacher/commitlint/generate | 生成预览 | — | 两个配置文件内容 |
| GET | /api/teacher/commitlint/branches | 分支列表 | — | {branches, default_branch} |
| POST | /api/teacher/commitlint/push | 推送仓库 | branch 参数 | {commit_sha, commit_url} |
| POST | /api/teacher/commitlint/save-and-push | 保存并推送 | 规则 + branch | {commit, version, ...} |
| GET | /api/teacher/commitlint/repo-status | 仓库状态 | — | {token_ok, file_exists, ...} |

```json

  "enabled": true,
  "rule_type": "conventional",

  "header_max_length": 100,
```

## 5. 技术选型说明

### 5.1 配置文件生成引擎（commitlint_engine.py）

核心函数 `build_commitlintrc()` 根据 `is_enabled` 参数动态调整规则级别：

- **enabled=True** → 规则 level=2（error，CI 阻断）
- **enabled=False** → 规则 level=0（关闭，CI 跳过）

生成的配置文件包含三条核心规则：
- `type-enum`：校验 commit type 是否在允许列表中
- `header-max-length`：校验 header 不超过最大长度
- `subject-min-length`：校验 subject 不少于最小长度

### 5.2 GitHub API 选型

| API 方案 | 所需权限 | 复杂度 | 选择 |
|---------|---------|--------|------|
| Git Data API（Blob→Tree→Commit→Refs） | Contents: Write | 高（4 步） | ❌ |
| **Contents API（PUT /contents/{path}）** | **Contents: Write** | **低（1 步）** | **✅** |

采用 Contents API 的理由：
1. 单步操作即可完成文件创建/更新
2. 权限需求最小
3. 自动处理 Base64 编码
