## 2. 编码规范

### 2.1 遵循标准

- **PEP 8**：Python 代码风格指南
- **PEP 484**：类型注解（函数参数和返回值类型标注）

### 2.2 命名规范

| 元素 | 规范 | 代码示例 |
|------|------|---------|
| 文件名 | 蛇形命名 | `commitlint_engine.py`, `github_client.py` |
| 类名 | 帕斯卡命名 | `CommitlintState`, `CommitlintConfigUpdate` |
| 函数/方法 | 蛇形命名 | `verify_teacher_token()`, `build_commitlintrc()` |
| 常量 | 大写蛇形 | `GIT_BRANCH`, `MAX_FAILURES` |
| 私有方法 | 下划线前缀 | `_require_teacher()`, `_load_config()` |

### 2.3 代码风格约定

- 字符串统一使用双引号
- 行尾不保留多余空格
- 每行不超过 100 个字符
- 函数和方法需声明类型注解
- 私有函数和方法以下划线 `_` 开头

## 3. 关键模块说明

### 3.1 commitlint_engine（核心引擎）

| 属性 | 说明 |
|------|------|
| **职责** | 生成 `.commitlintrc.json` 和 `.github/workflows/commitlint.yml` |
| **对外接口** | `build_commitlintrc(**kwargs)` → JSON 字符串；`generate_workflow_yml()` → YAML 字符串 |
| **被谁调用** | `routers/commitlint.py` 和 `devops_commitlint.py` |
| **核心逻辑** | `is_enabled=True` → rules level=2；`is_enabled=False` → level=0 |

### 3.2 github_client（GitHub API 客户端）

| 属性 | 说明 |
|------|------|
| **职责** | 封装 GitHub REST API 调用，提供文件推送和仓库查询 |
| **对外接口** | `push_files()`、`push_file()`、`check_token()`、`file_exists()`、`get_file_sha()`、`list_branches()` |
| **依赖** | `GITHUB_TOKEN` 和 `GITHUB_REPO` 环境变量 |
| **API 方案** | 使用 Contents API（`PUT /repos/{owner}/{repo}/contents/{path}`）逐文件推送 |
| **编码方式** | Base64 编码文件内容 |
| **错误处理** | HTTPError 统一捕获并包装为 RuntimeError |

### 3.3 commitlint.py 路由

| 属性 | 说明 |
|------|------|
| **职责** | 处理所有 F-D-009 的 HTTP 请求 |
| **API 数量** | 10 个接口（GET 5 个，POST 4 个，PUT 1 个） |
| **请求校验** | Pydantic 模型（`CommitlintConfigUpdate`、`SaveAndPushRequest`） |
| **认证** | `_require_teacher()` 统一校验教师 Token |
| **关键函数** | `ensure_mysql_row()` 启动时自动创建默认配置行 |

### 3.4 failures_store（失败记录存储）

| 属性 | 说明 |
|------|------|
| **职责** | 存储最近 5 条 commitlint 校验失败记录 |
| **存储方式** | 本地 JSON 文件（`<project_root>/.local_exam_data/commitlint_failures.json`） |
| **最大记录** | 5 条（新记录插入到头部，超出截断） |
| **数据结构** | `{id, commit_sha, commit_msg, author, branch, pr_number, failed_at, error_msg}` |

## 6. Git 工作流

### 6.1 Commit Message 格式（本功能自身使用的规范）

本功能强制要求团队成员遵循 Conventional Commits 规范：

```
<type>(<scope>): <description>

# type: feat | fix | chore | docs | refactor | test | style
# scope: 修改的模块名（可选）
# description: 中文或英文描述
```

### 6.2 F-D-009 开发过程中的 Commit 示例

```
feat(commitlint): 完成配置向导 API 基础框架
feat(commitlint): 实现配置文件生成引擎
feat(commitlint): 集成 GitHub Contents API 推送
feat(commitlint): 前端添加分支选择器
fix(commitlint): 修复 MySQL 外键约束初始化失败
refactor(commitlint): 将 Git Data API 改为 Contents API
test(commitlint): 添加 TDD 测试 5 个用例
docs(commitlint): 完善工程实践文档
```
