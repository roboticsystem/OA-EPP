## 2. 技术收获

### 2.1 FastAPI 框架

通过本次项目，我深入掌握了 FastAPI 的核心特性：

- **路由模块化**：学习了如何使用 `APIRouter` 将 commitlint 的 10 个接口组织在独立文件中，再通过 `main.py` 的 `app.include_router()` 注册。这种模块化方式使代码结构清晰，各功能域互不干扰。
- **Pydantic 请求校验**：使用 `CommitlintConfigUpdate` 和 `SaveAndPushRequest` 两个 Pydantic 模型自动校验请求参数，配合 `@field_validator` 实现了 header_max_length 必须为正整数、rule_type 必须属于枚举值等校验逻辑。任何非法输入在进入业务逻辑之前就被拦截，大大减少了异常处理代码。
- **依赖注入**：通过 `Header(None)` 提取 `Authorization` 头，再用 `_require_teacher()` 统一校验教师身份，避免了在每个路由方法中重复编写 Token 解析代码。

### 2.2 GitHub API 集成

这是本次项目最大的技术挑战之一：

- **API 方案选型**：最初使用 Git Data API（Blob → Tree → Commit → Refs 四步流程），但遇到了 403 权限不足的问题。经过排查，改为使用 Contents API（单步 `PUT /contents/{path}`），只需要 `Contents: Write` 一个权限，操作也更为简洁。
- **Token 权限管理**：了解了 Fine-grained PAT 和 Classic Token 的区别，以及组织仓库的 SSO 授权流程。Classic Token 勾选 `repo` scope 即可访问组织仓库，但需要注意是否有 SAML SSO 限制。
- **错误处理和调试**：学会了区分 HTTP 403（权限不足）、404（仓库不存在/无访问权）、422（参数错误）等不同错误码，并通过 `check_token()` API 快速诊断 Token 有效性。

### 2.3 数据库设计与集成

- **双数据源架构**：学习了 SQLite（原业务数据库）与 MySQL（commitlint_configs 专用）的双数据源设计，通过 `database.py` 和 `mysql_db.py` 分离，环境变量控制切换。
- **列名映射**：MySQL 列名使用 `snake_case`（如 `header_max_len`），而前端 API 使用更友好的命名（如 `header_max_length`）。通过 `_FRONTEND_TO_MYSQL` 字典实现了双向映射，这是一个简洁且可扩展的设计模式。
- **外键约束处理**：`commitlint_configs` 引用了 `courses` 和 `teachers` 表，但这两个表在测试环境中可能为空。通过 `SET foreign_key_checks=0/1` 在初始化的临时绕过，既保证了数据完整性，又解决了初始化问题。

### 2.4 前端交互设计

- **异步数据流**：前端 commitlint 面板的 JavaScript 函数通过 `fetch()` + `async/await` 实现异步数据加载，每个操作后自动刷新状态总览和仓库状态。
- **状态驱动 UI**：所有按钮都有 loading 状态（禁用 + 文字提示），操作结果通过 toast 提示反馈，符合 Nielsen 可用性原则。
- **分支选择器**：通过 `GET /branches` 从 GitHub API 动态拉取分支列表渲染到 `<select>` 下拉框，用户选择后在后续 push/save-and-push 操作中自动携带。

### 2.5 TDD 测试驱动开发

- 理解了 TDD 的 RED → GREEN → REFACTOR 循环的真正含义：先写测试（RED），再实现代码（GREEN），最后优化（REFACTOR）。
- 5 个 TDD 测试用例覆盖了 State 属性、方法存在性、枚举验证、长度限制、容量限制，验证了核心逻辑的完整性。
- 测试用例编号 `TC01-TC05` 与需求 F-D-009 对应，便于追溯。

### 3.2 协作经验

## 4. AI 辅助工具使用总结

### 4.1 使用场景

本次项目我使用了 Trae IDE 中的 AI 辅助功能，主要在以下场景：

| 场景 | 使用方式 | 效果 |
|------|---------|------|
| **API 代码生成** | 描述需求后 AI 生成 FastAPI 路由代码 | 约 3x 效率提升 |
| **Bug 修复** | 粘贴错误信息，AI 定位根因并给出修复方案 | 显著缩短调试时间 |
| **GitHub API 集成** | 从 Git Data API 迁移到 Contents API | AI 对比两种方案优劣 |
| **文档编写** | 生成 Markdown 文档结构和内容 | 约 5x 效率提升 |

### 4.2 对 AI 生成代码的审查经验

AI 生成代码并非完美，我在使用过程中特别注意以下审查点：

1. **错误处理完整性**：AI 生成的代码可能忽略边界情况（如空列表、None 值），需要手动补充校验逻辑
2. **命名一致性**：AI 生成的变量名可能与项目现有风格不一致，需要统一调整
3. **安全合规**：AI 不会主动使用参数化查询和环境变量，需要人工检查和修正
4. **不过度设计**：AI 倾向于生成过度抽象的代码（不必要的工厂模式、基类等），遵循 YAGNI 原则保持简单
